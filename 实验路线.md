# WiFi DG 实验完整路线（适合 VSCode 阅读）

## 0. 总目标

本课题的核心目标是：

> 在 WiFi CSI 活动识别的 Domain Generalization（DG）设定下，系统比较  
> `ERM`、`IRM/REx`、`SparseIRM/SparseREx` 的目标域泛化能力，  
> 并分析性能变化究竟来自：
> 1. DG penalty（不变性约束）
> 2. sparse 稀疏化
> 3. 二者的组合效果

最终需要回答 4 个核心问题：

1. 一个强而稳定的 `ERM baseline` 能做到多好？
2. 在相同 backbone 下，`IRM / REx` 是否比 `ERM` 更好？
3. 单纯引入 `sparse` 是否有帮助？
4. `SparseIRM` 是否比 `IRM-Dense` 和 `ERM-Sparse` 更强？

---

## 1. 实验总原则

整个项目必须严格遵守以下原则：

- 所有方法使用 **同一套数据划分**
- 所有方法使用 **同一套输入预处理**
- 所有方法使用 **同一 backbone 拓扑**
- 所有方法使用 **同一 optimizer / scheduler / 训练轮数 / 选模规则**
- 唯一允许变化的因素只有：
  - `backbone_type = dense | sparse`
  - `penalty_type = none | irm | rex`

也就是说，实验设计必须是“单因素逐步变化”，避免多个因素同时改变导致结论无法解释。

## 2. 数据协议固定

### 2.1 数据集
当前阶段只支持：

- `csi_bench_activity`

### 2.2 输入格式
原始单样本 shape：

- `(500, 232)`

必须强制 reshape 为：

- `(4, 500, 58)`

解释固定为：

- `4`：MIMO 链路数
- `500`：时间步
- `58`：子载波数

进入模型前统一转为：

- `[B, 4, 500, 58]`

### 2.3 分类与环境设定
固定：

- `num_classes = 8`
- 源域环境：`env0, env1, env2, env3`
- 目标域环境：`env4`

### 2.4 数据划分协议
固定如下：

- `train`：来自源域 `env0-3`
- `val`：来自源域 `env0-3`
- `test`：来自未见目标域 `env4`

必须满足：

- `val` 中不能出现 `env4`
- `test` 中只能出现 `env4`
- 绝不允许用 `test` 做 model selection

### 2.5 归一化规则
归一化统计量只能从训练集计算：

- 在 `train` 上计算 mean/std
- 保存为缓存文件
- `val/test` 复用该统计量

禁止：

- 用全数据统计量
- 给 train/val/test 各自单独归一化

否则属于数据泄漏。

## 3. 实验框架阶段划分

整个实验路线分为 5 个阶段：

1. **阶段 A：底座搭建**
2. **阶段 B：建立强 ERM baseline**
3. **阶段 C：验证 IRM / REx 是否有收益**
4. **阶段 D：验证 sparse 本身是否有收益**
5. **阶段 E：验证 SparseIRM 是否有组合增益**

---

## 4. 阶段 A：先搭好统一实验底座

这一阶段不要急着追求最高精度，目标是：

> 保证实验系统正确、可重复、可扩展

### 4.1 需要完成的模块

#### 4.1.1 数据模块
实现 `wifi_dg/data.py`，完成：

- 读取 `train.npz / val.npz / test.npz`
- reshape `(500,232) -> (4,500,58)`
- 按训练集统计量归一化
- 返回 `(x, y, env)`

加入强校验：

- shape 错误直接报错
- 类别数不是 8 报错
- `train/val` 中出现 `env4` 报错
- `test` 中出现非 `env4` 报错
- 数据中出现 `NaN/Inf` 报错

#### 4.1.2 模型模块
实现统一的 `WiFiDGBackbone`：

结构建议：

- `stem`
- `3 个 CSI conv block`
- `adaptive pooling`
- `classifier`

设计原则：

- 输入固定 `[4,500,58]`
- 前几层尽量保持时间维分辨率
- 优先使用非对称卷积核：
  - `(5,3)`
  - `(7,3)`
  - `(3,5)`
- `classifier` 始终保持 dense
- `sparse` 只作用于 backbone

#### 4.1.3 训练模块
实现统一 trainer，支持：

- `penalty_type = none`
- `penalty_type = irm`
- `penalty_type = rex`

要求：

- 只能有一套训练主循环
- 不能复制出多套 trainer
- 方法差异必须由配置驱动

#### 4.1.4 评估模块
统一输出指标：

- `overall_acc`
- `macro_f1`
- `balanced_acc`
- `per_class_f1`
- `confusion_matrix`

对于 `val`，额外输出：

- `per_env_acc`
- `worst_source_env_acc`

#### 4.1.5 checkpoint 选择模块
best checkpoint 固定按以下规则排序：

1. `val_macro_f1`
2. `worst_source_env_acc`
3. `val_acc`

这个规则一旦确定，后续所有实验不能变。

#### 4.1.6 fast_dev_run
增加快速调试模式：

- `fast_dev_run = true`
- 或 `max_batches = 2`

用于快速验证：

- 数据流是否正确
- penalty 是否能算
- 日志是否正常
- checkpoint 是否正常保存
- eval 是否正常输出

## 5. 阶段 B：先做强 ERM baseline

这一阶段最关键。

核心目标：

> 先得到一个强、稳定、可复现的 `ERM-Dense baseline`

这是后续所有 DG 方法的参照标准。

### 5.1 方法定义
固定配置：

- `backbone_type = dense`
- `penalty_type = none`

### 5.2 实验顺序

#### Step 1：先做 smoke test
先用：

- `fast_dev_run = true`
- 或 `max_batches = 2`

确认：

- 训练能跑通
- loss 正常下降
- eval 正常
- best checkpoint 逻辑正常

#### Step 2：单 seed 正式训练
先只跑一个 seed，例如：

- `seed = 0`

要观察：

- train loss 是否正常下降
- val macro_f1 是否有提升
- 是否出现明显过拟合
- test 是否能正常评估

如果这一步都不稳定，不要进入 IRM。

#### Step 3：ERM 小范围调参
只在 ERM 上调参，因为 DG 方法更难训，必须先把 baseline 调强。

建议搜索参数：

- `learning_rate`：`1e-3`, `5e-4`, `2e-4`, `1e-4`
- `batch_size`：`16`, `32`, `64`
- `weight_decay`：`1e-4`, `5e-4`, `1e-3`
- `dropout`：`0.0`, `0.2`, `0.3`, `0.5`
- `epochs`：`30`, `50`, `80`
- `scheduler`：`cosine`, `step`, `none`

选参原则：

- 只能根据 `val` 选配置
- 不能根据 `test` 选配置

#### Step 4：固定最优 ERM 配置
当你找到最优 ERM 参数后，这套配置将成为后续方法的公共配置基线。

#### Step 5：跑 3 seeds 正式结果
固定配置后，运行：

- `seed = 0`
- `seed = 1`
- `seed = 2`

记录：

- `val best metrics`
- `test metrics`
- `mean ± std`

### 5.3 阶段 B 的产出
你必须拿到：

- 一个可信的 `ERM-Dense mean ± std`
- 一个稳定的训练配置
- 一组可复现实验结果

### 5.4 阶段 B 通过标准
满足以下条件后，才能进入下一阶段：

- ERM 能稳定收敛
- 多 seed 波动可接受
- test 结果可信
- 没有明显代码 bug 或数据泄漏

---

## 6. 阶段 C：验证 IRM / REx 是否有收益

这一阶段的目标是：

> 在完全相同的 dense backbone 上，只改变 penalty，验证 DG penalty 是否真的有帮助

### 6.1 实验 1：IRM-Dense

配置：

- `backbone_type = dense`
- `penalty_type = irm`

其余全部与 `ERM-Dense` 保持一致：

- 相同数据流
- 相同 backbone 拓扑
- 相同 optimizer
- 相同 scheduler
- 相同 epochs
- 相同选模规则

只能改变 penalty。

### 6.2 IRM 额外调参
IRM 一般不能完全照搬 ERM 参数，需要额外调以下参数：

- `penalty_weight`
- `penalty_anneal_epochs`

建议搜索范围：

- `penalty_weight`：`1e-2`, `1e-1`, `1`, `10`
- `penalty_anneal_epochs`：`0`, `5`, `10`, `20`

原因：

- 如果一开始 penalty 太强，模型可能学不到判别特征
- warmup / annealing 往往能显著提高稳定性

### 6.3 实验顺序
和 ERM 一样：

1. `fast_dev_run`
2. 单 seed 调通
3. 小范围调 penalty 参数
4. 固定配置
5. 跑 3 seeds

### 6.4 主要比较对象
比较：

- `ERM-Dense vs IRM-Dense`

重点关注：

- `test overall_acc`
- `test macro_f1`
- `balanced_acc`
- `val worst_source_env_acc`
- 结果方差是否变大

---

### 6.5 实验 2：REx-Dense（推荐）
由于很多任务中 `REx` 往往比原始 `IRM` 更稳，所以建议也做：

配置：

- `backbone_type = dense`
- `penalty_type = rex`

额外调参同样包括：

- `penalty_weight`
- `penalty_anneal_epochs`

### 6.6 阶段 C 的目标
这一阶段需要明确回答：

- DG penalty 在 dense backbone 上是否真的有收益？
- `IRM` 和 `REx` 哪个更稳？
- 如果 IRM 没提升，REx 是否有提升？

### 6.7 阶段 C 通过标准
满足以下条件后再进入 sparse 阶段：

- 已经明确 dense backbone 下 penalty 是否有效
- 已经大致知道 penalty weight / anneal 该怎么设
- 训练过程没有明显崩坏

如果 dense 上 `IRM/REx` 完全没收益，继续上 SparseIRM 时要格外谨慎。

---

## 7. 阶段 D：验证 sparse 本身是否有收益

这一阶段的目标是：

> 在不加 DG penalty 的情况下，只把 backbone 改成 sparse，判断 sparse 自身是否有帮助

### 7.1 方法定义
配置：

- `backbone_type = sparse`
- `penalty_type = none`

### 7.2 公平性要求
这里只允许改变 backbone 的层实现，不允许改变整体拓扑：

- block 数不变
- 通道数不变
- pooling 不变
- classifier 不变

否则对比不公平。

### 7.3 实验顺序
和前面一样：

1. `fast_dev_run`
2. 单 seed
3. 只调必要的 sparse 相关参数
4. 固定配置
5. 跑 3 seeds

### 7.4 主要比较对象
比较：

- `ERM-Dense vs ERM-Sparse`

关注：

- test 性能是否提升或下降
- val 是否更稳
- 稀疏率是多少
- active params 变化多少

### 7.5 额外必须记录的指标
`sparse` 实验不能只记录分类性能，必须同时记录：

- `global sparsity`
- `layer-wise sparsity`
- `active params`
- 可选：`approx FLOPs`

否则无法分析 sparse 到底起了什么作用。

### 7.6 阶段 D 目标
明确回答：

- 单纯 sparse 是否有帮助？
- sparse 是在压缩模型还是在提升泛化？
- sparse 对 WiFi 特征学习是正面还是负面作用？

---

## 8. 阶段 E：验证 SparseIRM / SparseREx

这是你的主方法阶段。

目标是：

> 验证 sparse + DG penalty 是否具有组合增益

### 8.1 主方法：SparseIRM
配置：

- `backbone_type = sparse`
- `penalty_type = irm`

### 8.2 可选扩展：SparseREx
配置：

- `backbone_type = sparse`
- `penalty_type = rex`

### 8.3 调参原则
不要重新做全量大搜索。

建议：

- 继承 `ERM-Sparse` 的主体训练参数
- 只额外调整：
  - `penalty_weight`
  - `penalty_anneal_epochs`

### 8.4 关键比较对象
至少要比较以下几组：

1. `ERM-Dense vs SparseIRM`
2. `IRM-Dense vs SparseIRM`
3. `ERM-Sparse vs SparseIRM`

如果做了 `SparseREx`，还可以比较：

4. `REx-Dense vs SparseREx`
5. `ERM-Sparse vs SparseREx`

### 8.5 阶段 E 要回答的问题
这一阶段必须明确回答：

- 相比基础 ERM，SparseIRM 是否有明显提升？
- 相比 dense IRM，sparse 是否进一步提升？
- 相比 sparse ERM，DG penalty 是否带来额外收益？
- sparse 和 IRM 是否真的存在组合效应？

一旦固定，不再随意改动。

### Step 5：3 seeds 正式结果
运行：

- `seed=0`
- `seed=1`

- `mean ± std`

## 14. 每个阶段的“是否可以继续”判断标准

### 14.1 是否可以从 ERM 进入 IRM
只有在以下条件满足时才继续：

- ERM 稳定收敛
- 多 seed 波动可接受
- test 结果可信
- 没有数据泄漏或明显 bug

### 14.2 是否可以从 dense DG 进入 SparseIRM
只有在以下条件满足时才继续：

- 已明确 `IRM/REx` 是否有效
- penalty 参数范围已有概念
- 训练过程没有明显崩坏

如果 dense 上 `IRM` 完全无收益，直接上 SparseIRM 需要非常谨慎。

## 15. 最容易犯的错误

### 错误 1：一开始就直接跑 SparseIRM
这样结果即使变化了，也没法解释是 sparse 还是 IRM 导致的。

### 错误 2：ERM baseline 太弱
如果 baseline 没调好，后面方法提升没有说服力。

### 错误 3：不公平比较
例如：

- ERM 用一个 backbone
- SparseIRM 用另一个更强 backbone

这样结论不成立。

### 错误 4：用 test 选模型
这会直接破坏 DG 实验的有效性。

### 错误 5：不做多 seed
单次结果波动太大，结论不可靠。

---

## 16. 最简执行顺序（真正开始时照着做）

### 第一批：必须先做
1. 跑通 `ERM-Dense`
2. 调出强 baseline
3. 跑 3 seeds
4. 固定 baseline 配置

### 第二批：验证 DG penalty
5. 跑 `IRM-Dense`
6. 如果 IRM 不稳，再跑 `REx-Dense`
7. 比较 DG penalty 是否真的有效

### 第三批：验证 sparse
8. 跑 `ERM-Sparse`
9. 记录稀疏率和性能变化

### 第四批：做主方法
10. 跑 `SparseIRM`
11. 可选跑 `SparseREx`
12. 做完整对比和消融

---

## 17. 一句话总结

整个实验路线的核心是：

> 先把 `ERM-Dense` 做成一个强而稳定的基线，  
> 再在完全相同条件下依次验证  
> `IRM/REx 是否有用`、`sparse 是否有用`、`SparseIRM 是否有额外组合增益`，  
> 最终用多 seed、公平对照和稀疏统计支撑结论。
